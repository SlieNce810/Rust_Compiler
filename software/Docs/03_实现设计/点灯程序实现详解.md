# 点灯程序实现详解（基于本项目 Rust 编译器）

本文档说明：

1. 这次“点灯 demo”是如何在本项目里实现的。
2. 从 `.mc` 源码到 `.asm` 输出的完整路径。
3. 为什么当前用 GPIO 占位接口变量也能先跑通流程。
4. 你后续如何替换为真实开发板 GPIO 接口。

---

## 1. 目标与边界

### 1.1 目标

在不依赖外部 C 工程的前提下，使用当前 Rust 编译器完成：

- `STM32` 点灯 demo 源码
- `ESP32` 点灯 demo 源码
- 分别编译输出目标风格的汇编文本

### 1.2 当前边界

当前编译器后端是“教学/模板级后端”，可以：

- 做词法、语法、语义检查
- 生成三地址码（TAC）
- 按 `stm32f403` / `esp32` 输出不同风格汇编

当前还不能直接产出可烧录固件（缺完整 ABI、链接脚本、真实寄存器分配等）。

所以这次点灯实现分为两层：

- **逻辑层已完整**：初始化 GPIO、翻转 LED、软件延时
- **硬件层先占位**：GPIO 写法先用占位变量，后续替换为真实接口

## 3. 点灯程序逻辑设计

两个平台版本逻辑完全一致，仅变量名区分：

1. 定义状态变量：
- `is_running`：主循环开关
- `led_state`：LED 当前电平（0/1）
- `delay_counter`：延时计数器
- `delay_limit`：延时长度

2. 初始化：
- 设置 `is_running = true`
- 设置 `led_state = 0`
- 设置 `delay_limit = 5000`
- 执行 GPIO 方向配置（占位变量赋值）

3. 主循环：
- `if (led_state == 0)` 切到 `1`，否则切到 `0`
- 将 `led_state` 写到 GPIO 输出（占位变量赋值）
- 执行空转延时循环

4. 返回：
- 代码结构要求返回 `int`，返回 `0`

## 4. `.mc` 源码如何映射到“点灯行为”

### 4.1 STM32 版本关键变量

- `stm32_gpio_mode_reg`：占位，表示 GPIO 模式寄存器配置
- `stm32_gpio_output_reg`：占位，表示 GPIO 输出寄存器写电平

核心代码片段（语义）：

- `stm32_gpio_mode_reg = 1;`：把 LED 引脚设为输出模式（占位）
- `stm32_gpio_output_reg = led_state;`：把 LED 电平写出去（占位）

### 4.2 ESP32 版本关键变量

- `esp32_gpio_enable_reg`：占位，表示 GPIO 输出使能配置
- `esp32_gpio_output_reg`：占位，表示 GPIO 输出寄存器写电平

核心代码片段（语义）：

- `esp32_gpio_enable_reg = 1;`：使能为输出（占位）
- `esp32_gpio_output_reg = led_state;`：写电平（占位）

## 5. 编译链路：从源码到汇编

当前工程主流程在 `compiler/src/main.rs`：

1. 读入 `.mc` 源文件
2. `lexer::tokenize`：词法分析
3. `parser.parse_program`：语法分析，生成 AST
4. `semantic::check_program`：语义检查
5. `ir::build_three_address_code`：生成 TAC
6. `backend::build_assembly_code`：输出目标汇编文本
7. 写入 `.asm`

这条链路对 `main.mc` 和两个点灯 demo 完全一致，说明 demo 与编译器是同一工作路径，不是“旁路脚本”。

---

## 6. 后端如何处理点灯循环

以 `while (is_running) { ... }` 为例，后端会生成：

1. 循环头标签（例如 `L0`）
2. 条件判断，条件为假时跳到循环结束标签
3. 循环体代码（翻转 LED + 延时）
4. 无条件跳回循环头

以 `if (led_state == 0)` 为例，后端会生成：

1. 比较结果临时值
2. 条件跳转到 `else` 标签
3. then 分支
4. 跳转到 end 标签
5. else 分支
6. end 标签

这就是你在输出汇编里看到一组 `Lx` 标签和 `b/j/beq/beqz` 的原因。

---

## 7. 栈帧与局部变量（为什么会看到 ldr/str 或 l32i/s32i）

后端已经实现了基础栈帧：

- 函数入口生成 prologue
- 局部变量分配栈偏移
- 读变量先 load 到 scratch 寄存器
- 写变量再 store 回栈
- return 统一跳到 epilogue

所以你会在点灯汇编里看到：

- STM32：`push/mov fp/sub sp/ldr/str/pop`
- ESP32：`entry/l32i/s32i/addi/ret`

这代表 demo 不是单纯文本替换，而是经过了“变量落栈”这一层后端逻辑。

---

## 8. 为什么现在用 GPIO 占位变量

原因很直接：

1. 当前自定义语言还不支持“内存映射地址写入”语法和外部函数调用语法。
2. 不同开发板的 GPIO 寄存器地址、时钟使能、HAL API 都不一样。
3. 先用占位变量能把编译链路先跑通，确保“语言-编译器-后端”都稳定。

这避免了在编译器功能未完备时，过早绑死某一块板子的寄存器细节。

## 9. 后续替换真实 GPIO 的建议路径

### 9.1 先替换初始化

- STM32：将 `stm32_gpio_mode_reg = 1;` 替换为真实 GPIO 模式配置
- ESP32：将 `esp32_gpio_enable_reg = 1;` 替换为真实 GPIO 方向配置

### 9.2 再替换输出

- STM32：将 `stm32_gpio_output_reg = led_state;` 替换为真实写引脚电平
- ESP32：将 `esp32_gpio_output_reg = led_state;` 替换为真实写引脚电平

### 9.3 最后替换延时

将软件空转延时替换为板级时基（如 SysTick / RTOS delay）。

这样做的顺序可以保证每一步都可单独验证，出问题好定位。
