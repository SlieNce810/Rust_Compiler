# Rust_Compiler 端侧解释器自顶向下总学习笔记

> 适用人群：第一次接触本项目端侧 VM 的同学（含小白）  
> 阅读方式：先读第 0 章学习地图，再按章节顺序从“主机到端侧”走一遍  
> 这份笔记的改进：统一源码绑定、补全调用链、强制分层标记（必懂/扩展/可跳过）

## 学习前：你需要知道的基础

- Rust 基础（必须会）：能看懂 `main.rs` 的编译入口和参数处理
- C 基础（必须会）：能看懂 `switch(opcode)` 执行循环
- MCU 基础（不会也没关系）：GPIO/UART 的最小概念
- 编译原理基础（建议）：知道 IR 与字节码分别是什么

## 第 1 章：主机编译产物层（必懂）

### 1) 它在整个 OS 里的位置（宏观）

这是“输入程序到可执行协议”的网关层，像快递分拣中心：上游收源码，下游交 `.hbc`。

本章目标：搞清楚 `main.hbc` 是怎么从 `main.hopping` 生成的。

### 2) 在本项目中如何实现（源码绑定）

文件列表：

| 文件 | 作用 |
| --- | --- |
| `software/compiler/src/main.rs` | 编译入口与参数分发 |
| `software/compiler/src/semantic.rs` | V1 子集约束 |
| `software/compiler/src/ir.rs` | IR 生成 |
| `software/compiler/src/bytecode.rs` | IR 转 HBC1 字节码 |

完整调用链：

`main.rs:run_compile_flow()`  
`-> compile_source()`  
`-> semantic::check_program()`  
`-> ir::build_three_address_code()`  
`-> bytecode::build_bytecode()`  
`-> 输出 software/examples/bytecode/main.hbc`

### 3) 关键机制拆解

机制 A（必懂）：固定演示产物路径  
代码路径：`software/compiler/src/main.rs`

```rust
// file: software/compiler/src/main.rs
if options.emit_demo_artifacts {
    save_text_file(&PathBuf::from("../examples/ir/main.ir"), &compile_output.ir_text)?;
    save_binary_file(&PathBuf::from("../examples/bytecode/main.hbc"), &compile_output.bytecode)?;
}
```

> **名词解释：artifact**  
> 编译产物文件（如 `.ir`、`.hbc`）。

**关键发现：固定产物路径能稳定对接两端固件流程。**

机制 B（必懂）：V1 子集前置拒绝  
代码路径：`software/compiler/src/semantic.rs`

```rust
// file: software/compiler/src/semantic.rs
if program.function_list.len() != 1 {
    return Err("V1 subset: only one function is allowed (func main)".to_string());
}
```

> **名词解释：subset（子集）**  
> 为了控制复杂度，先只允许语法/语义的一部分能力。

**关键发现：在语义层拒绝非法输入，比端侧崩溃更可控。**

### 4) 小白理解检查

Q：这一层在解决什么问题？  
A：把“人写的源码”变成“MCU 能执行的统一格式字节码”。

Q：如果不用会怎样？  
A：端侧就需要直接理解源码，复杂度和资源开销都会失控。

### 5) 知识补充（如果需要）

IR 是“中间语言”，目的是让后端更容易处理复杂表达式，不是最终运行格式。

### 6) 本节总结

- 主机侧先负责把源码压缩成可执行协议
- V1 子集是为了端侧可控落地
- 产物路径固定后，真机流程才稳定

## 第 2 章：字节码协议层（必懂）

### 1) 它在整个 OS 里的位置（宏观）

协议层是主机和端侧之间的“合同”，双方都按这份合同读写数据。

本章目标：理解 `HBC1` 文件结构和指令编码。

### 2) 在本项目中如何实现（源码绑定）

文件列表：

| 文件 | 作用 |
| --- | --- |
| `software/compiler/src/bytecode.rs` | 写出 HBC1 协议 |
| `software/firmware/common/vm_core.c` | 解析 HBC1 协议 |
| `software/Docs/03_实现设计/MCU解释器V1设计.md` | 协议说明文档 |

完整调用链：

`build_bytecode()`  
`-> 写 header/symbol/instruction`  
`-> main.hbc`  
`-> vm_load_program()`  
`-> parse_header()/parse_symbols()`

### 3) 关键机制拆解

机制 A（必懂）：header 校验  
代码路径：`software/firmware/common/vm_core.c`

```c
/* file: software/firmware/common/vm_core.c */
if (buffer[0] != VM_MAGIC_0 || buffer[1] != VM_MAGIC_1 || buffer[2] != VM_MAGIC_2 || buffer[3] != VM_MAGIC_3) {
    return VM_ERR_BAD_MAGIC;
}
if (buffer[4] != VM_VERSION) return VM_ERR_BAD_VERSION;
```

> **名词解释：magic**  
> 文件签名，用于快速识别文件类型。

**关键发现：header 是端侧第一道防线，错了立即拒绝。**

机制 B（必懂）：12 字节指令解码  
代码路径：`software/firmware/common/vm_core.c`

```c
/* file: software/firmware/common/vm_core.c */
inst.opcode = p[0];
inst.dst = p[1];
inst.lhs = p[2];
inst.rhs = p[3];
inst.mode = p[4];
inst.target = read_u16(&p[6]);
inst.imm = read_i32(&p[8]);
```

> **名词解释：opcode**  
> 指令类型编号，决定执行哪个语义分支。

**关键发现：编码布局一旦改动，主机和端侧必须同步升级。**

### 4) 小白理解检查

Q：协议层解决什么？  
A：保证“同一份字节码”在不同芯片上被同样理解。

Q：不用协议会怎样？  
A：主机产物和端侧解析会错位，轻则报错，重则执行错逻辑。

### 5) 知识补充（如果需要）

LE（Little Endian）是小端序，低字节放前面；`read_u16/read_i32` 就是在按小端读。

### 6) 本节总结

- `HBC1` 是主机与端侧通信合同
- header 校验用于快速拒绝坏输入
- 指令布局必须保持双端一致

## 第 3 章：VM 加载与校验层（必懂）

### 1) 它在整个 OS 里的位置（宏观）

这是“门卫层”，负责在执行前筛掉坏字节码，避免执行阶段崩溃。

本章目标：读懂 `vm_load_program()` 全流程。

### 2) 在本项目中如何实现（源码绑定）

文件列表：

| 文件 | 作用 |
| --- | --- |
| `software/firmware/common/vm_core.c` | load + validate 主流程 |
| `software/firmware/common/vm_core.h` | 错误码与状态定义 |

完整调用链：

`vm_load_program()`  
`-> parse_header()`  
`-> parse_symbols()`  
`-> attach_instruction_view()`  
`-> validate_instruction_format()`  
`-> validate_jump_targets()`  
`-> clear_state()`

### 3) 关键机制拆解

机制 A（必懂）：指令格式预检  
代码路径：`software/firmware/common/vm_core.c`

```c
/* file: software/firmware/common/vm_core.c */
if (!is_mode_valid(inst->mode)) return VM_ERR_BAD_HEADER;
if (inst->reserved != 0) return VM_ERR_BAD_HEADER;
```

> **名词解释：reserved**  
> 预留位，V1 规定必须为 0，用于未来扩展兼容。

**关键发现：越早预检，越少把错误带入执行循环。**

机制 B（必懂）：跳转目标预检  
代码路径：`software/firmware/common/vm_core.c`

```c
/* file: software/firmware/common/vm_core.c */
if (inst.opcode == VM_OP_JMP || inst.opcode == VM_OP_JMP_IF_ZERO) {
    if (inst.target >= vm->program.instruction_count) return VM_ERR_BAD_JUMP_TARGET;
}
```

> **名词解释：target**  
> 跳转目标指令索引（PC 位置）。

**关键发现：jump 越界是 MCU 解释器最常见致命错误之一。**

### 4) 小白理解检查

Q：这层在解决什么？  
A：确保“要执行的程序”在结构上是合法的。

Q：如果省略这层？  
A：执行循环里会出现大量不可预测异常，定位成本极高。

### 5) 知识补充（如果需要）

`error_code` 不是日志文本，而是稳定枚举值；跨平台对拍要看码值是否一致。

### 6) 本节总结

- 加载阶段是防崩溃关键环节
- 预检覆盖 mode、寄存器、jump target
- 错误码统一是跨平台一致性的基础

## 第 4 章：VM 执行循环层（必懂）

### 1) 它在整个 OS 里的位置（宏观）

执行循环是“解释器发动机”，决定端侧实际行为。

本章目标：掌握 `while(!halted)` + `switch(opcode)` 的执行语义。

### 2) 在本项目中如何实现（源码绑定）

文件列表：

| 文件 | 作用 |
| --- | --- |
| `software/firmware/common/vm_core.c` | dispatch 主逻辑 |
| `software/firmware/common/vm_core.h` | opcode 定义 |

完整调用链：

`vm_run()`  
`-> fetch(read_instruction)`  
`-> dispatch(switch opcode)`  
`-> 更新 pc/regs/retval`  
`-> halt 或继续`

### 3) 关键机制拆解

机制 A（必懂）：`ifz` 统一语义  
代码路径：`software/firmware/common/vm_core.c`

```c
/* file: software/firmware/common/vm_core.c */
if (value == 0) vm->pc = inst.target;
else vm->pc++;
```

> **名词解释：ifz**  
> if zero，条件寄存器为 0 时跳转。

**关键发现：统一 `reg==0` 可避免跨平台“真值语义”漂移。**

机制 B（必懂）：`return` 停机语义  
代码路径：`software/firmware/common/vm_core.c`

```c
/* file: software/firmware/common/vm_core.c */
vm->retval = value_or_imm;
vm->halted = 1;
vm->pc++;
```

> **名词解释：retval**  
> 程序返回值，用于驱动上层逻辑（如点灯条件）。

**关键发现：`retval` 是业务输出，`error_code` 是运行状态，两者不能混用。**

### 4) 小白理解检查

Q：执行循环在解决什么？  
A：把“字节码描述”变成“真实寄存器状态变化和控制流变化”。

Q：如果实现不一致？  
A：同一 `main.hbc` 在两平台会跑出不同结果。

### 5) 知识补充（如果需要）

`step_limit` 是保险丝：防止死循环把设备卡死。

### 6) 本节总结

- dispatch 是解释器核心
- `ifz/return` 是行为一致性的关键点
- `step_limit` 是运行安全底线

## 第 5 章：HAL 抽象与平台层（扩展）

### 1) 它在整个 OS 里的位置（宏观）

HAL 是“硬件翻译层”：上层说“点灯/打日志”，下层变成具体芯片调用。

本章目标：明白为什么 VM 核心不直接写 GPIO/UART。

### 2) 在本项目中如何实现（源码绑定）

文件列表：

| 文件 | 作用 |
| --- | --- |
| `software/firmware/common/vm_hal.h` | 统一 HAL 接口 |
| `software/firmware/stm32f407/Core/Src/vm_hal_stm32.c` | STM32 实现 |
| `software/firmware/esp32s3/main/vm_hal_esp32.c` | ESP32 实现 |

完整调用链：

`main.c/app_main()`  
`-> hal_init()/hal_log_int()/hal_led_write()`  
`-> 平台 HAL 实现`  
`-> 外设驱动`

### 3) 关键机制拆解

机制（扩展）：统一日志键值  
代码路径：`software/firmware/common/vm_hal.h`

```c
/* file: software/firmware/common/vm_hal.h */
void hal_log_int(const char *key, int value);
```

> **名词解释：HAL（Hardware Abstraction Layer）**  
> 硬件抽象层，屏蔽不同芯片差异。

**关键发现：日志键统一后，跨平台行为对拍成本大幅降低。**

### 4) 小白理解检查

Q：为什么不直接在 VM 里调用芯片库？  
A：那样 VM 会被平台绑定，无法复用。

### 5) 知识补充（如果需要）

接口要“最小但稳定”，否则平台迁移会被接口变更拖慢。

### 6) 本节总结

- HAL 把平台差异隔离在边缘
- VM 核心保持纯逻辑
- 日志接口是双平台对拍基石

这是最终落地层，验证“设计是否真的能在板子上跑起来”。

本章目标：掌握 STM32F407 + ESP32S3 的统一验收口径。

> 用同一输入比较两平台输出是否一致。

Q：为什么要两平台都跑同一字节码？  
A：这是“跨平台”的最直接证据。

### 6) 本节总结

- 同一字节码是跨平台测试基准
- 验收先看语义一致，再看性能细节
- 文档化验收流程能防止“只在一块板子有效”

## 第 7 章：高频机制与易错点

### 跨章节共通机制表

| 机制 | 一句话解释 | 在哪里反复出现 |
| --- | --- | --- |
| Header 校验 | 程序合法性的第一道门 | 第 2/3 章 |
| 预检后执行 | 先拒绝坏包再运行 | 第 3/4 章 |
| `ifz` 统一语义 | 条件分支只认 `reg==0` | 第 2/4/6 章 |
| 错误码统一 | 跨平台定位问题的坐标系 | 第 3/6 章 |
| HAL 最小接口 | 核心与平台解耦 | 第 5/6 章 |

### 高频坑位表

| 坑位 | 表现 | 怎么避免 |
| --- | --- | --- |
| 修改指令布局未同步双端 | 端侧报 `BAD_HEADER/BAD_OPCODE` | 先改协议文档，再改主机和端侧 |
| 跳转目标越界 | 运行时报 `VM_ERR_BAD_JUMP_TARGET` | 保留加载阶段预检 |
| `retval` 与 `error_code` 混用 | 业务逻辑判断错误 | 日志分开打：`vm_retval` 与 `vm_run_err` |
| 平台日志键不一致 | 无法对拍 | 统一使用 `hal_log_int(key, value)` |

## 实战方法论

五步法：

1. 理解目标：本轮是“闭环可展示”，不是“功能全覆盖”
2. 规划：先协议，再 VM，再平台
3. 最小实施：先跑通 `main.hbc` 固件内置
4. 验证闭环：双平台同字节码对拍
5. 记录总结：把坑位和机制沉淀进文档

| IR | Intermediate Representation | 编译中间表示，便于后端处理 |
| opcode | operation code | 指令类型编号 |
| HAL | Hardware Abstraction Layer | 硬件抽象接口层 |
| PC | Program Counter | 当前指令位置 |
| retval | return value | 程序业务返回值 |
| error_code | error code | VM 执行状态错误码 |

## 一句话总复盘

先把 `Hopping` 逻辑稳定编译成统一 `HBC1`，再用同一 VM 语义在双平台执行并对拍日志，就是本项目端侧路线的主线闭环。
