# 建议汇总

## 方向判断

主机编译 `.mc`、设备执行轻量 IR/字节码、支持热更新，是当前阶段最稳妥且可跨平台的路线。  
不建议当前直接做“设备端完整编译器”。

## 跨平台策略

跨平台核心不在“移植整套编译器”，而在以下分层：

1. 主机侧统一：前端、语义检查、IR/字节码生成
2. 设备侧统一：解释器执行循环
3. 平台差异收敛在 HAL：GPIO/UART/Timer 等接口

## 实施顺序建议

1. 先打通串口闭环（最稳定）
2. 再做热更新安全机制（CRC、回滚、边界检查）
3. 最后接 BLE/WiFi 与手机 App（作为增强项）

## 解释器实现建议（MCU 重点）

设备端最小结构建议：

1. 程序区：存放字节码
2. VM 状态：`pc`、寄存器、标志位、资源表
3. 指令分发：`switch(opcode)` 执行
4. HAL 适配：所有硬件操作只走 `hal_*` 接口

建议第一版指令集只保留必要能力：

1. 数据指令：`LOAD_IMM`、`MOV`
2. 运算指令：`ADD`、`SUB`、`MUL`、`DIV`
3. 分支指令：`JMP`、`JMP_IF_ZERO`
4. 外设指令：`GPIO_WRITE/READ`、`UART_WRITE`、`TIMER_READY`
5. 控制指令：`HALT`、`RETURN`

## 安全规则建议

优先在编译期做强约束：

1. 资源先声明再绑定
2. 资源先配置再使用
3. 输入引脚禁止写，输出引脚禁止按输入读
4. 未启用外设禁止操作
5. 参数范围非法直接报错

## 竞赛视角建议

要拿分，重点展示“系统能力闭环”，而不是单点功能：

1. 同一套 `.mc` 逻辑可跨平台运行
2. 可热更新逻辑，无需重刷完整固件
3. 无线链路只是下载通道，不是唯一亮点

## 执行规范

后续每次任务统一流程：

1. 先说明怎么做
2. 再说明为什么
3. 再说明用什么
4. 然后执行并验证结果
