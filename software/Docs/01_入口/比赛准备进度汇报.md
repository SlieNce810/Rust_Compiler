# 比赛准备进度汇报

这份文档面向导师汇报，目标是把当前工作说清楚，让项目方向、当前进度和下一步计划都能被快速理解。

## 1. 项目目标

本项目的核心目标不是单纯做一个编译器，而是做一套面向嵌入式的跨平台控制语言和运行系统。

当前目标可以概括成一句话：

设计一门跨平台嵌入式语言 `Hopping`，在主机侧编译，在设备侧通过轻量 IR / 字节码解释执行，实现逻辑热更新。

这条路线的意义是：

- 同一份逻辑可以面向不同硬件平台复用
- 不需要每次都重刷完整固件
- 后续可以扩展到串口、蓝牙、WiFi 等下载链路

## 2. 当前技术路线

当前已经明确采用这条路线：

1. 主机侧：
- 编写 `Hopping` 源码
- 做词法分析、语法分析、语义检查
- 生成 IR
- 生成字节码
- 可继续生成汇编用于调试和验证

2. 设备侧：
- 不放完整编译器
- 只放轻量解释器 / 运行时
- 接收主机下发的字节码
- 在 MCU 上执行逻辑

这比“把完整编译器移植到单片机上”更现实，也更符合比赛阶段的工程收敛。

## 4. 当前项目状态判断

目前项目已从“主机侧编译原型”进入“双平台端侧执行闭环”阶段。

当前阶段结论：

- 语言输入可控（V1 子集）
- 字节码格式稳定（HBC1 v1）
- 端侧解释器可运行（STM32 + ESP32）
- 展示闭环已形成（编译 -> 烧录 -> 解释执行 -> 点灯/日志）

## 6. 导师最关心的问题，应该怎么回答

### 6.1 “你这个项目的核心创新点是什么？”

建议回答：

不是单纯做一个编译器，而是做一套面向嵌入式的跨平台语言和轻量运行时系统。

核心点在于：

- 自定义嵌入式语言
- 主机侧编译
- 设备侧字节码执行
- 支持逻辑热更新
- 目标是跨 STM32 / ESP32 等平台复用

### 6.2 “为什么不直接写普通固件？”

建议回答：

普通固件开发每次修改都要重新编译和整包烧录，迭代成本高，且跨平台复用差。

`Hopping` 的目标是把“控制逻辑”从平台细节里抽离出来，让逻辑层更容易迁移和热更新。

### 6.3 “为什么不直接让 MCU 上跑编译器？”

建议回答：

设备资源有限，端侧完整编译器成本太高。

因此当前选择更务实的路线：

- 主机负责编译
- MCU 只负责解释执行字节码

这样更符合实际资源约束，也更容易形成稳定闭环。

### 6.4 “跨平台具体体现在哪里？”

建议回答：

跨平台不是指一份汇编在所有芯片通用，而是：

- 同一份 `Hopping` 逻辑
- 同一套主机编译链
- 同一套字节码格式
- 不同平台只实现各自的 HAL 和解释器适配

这样跨平台的代价被压缩到了硬件抽象层。

## 7. 当前还缺什么

目前最关键缺口已收敛到两类：

1. bootloader / 动态下载协议
2. 动态下发后的安全与回滚机制

解释器核心与双平台 Demo 闭环已完成，本阶段不再是“只有主机侧”。

## 8. 当前风险与控制

### 8.1 风险

- 过早并行做协议 + 无线 + 应用层，工期失控
- 动态下载上线后若校验策略不足，容易引入稳定性风险
- 不做一致性对拍，双平台行为可能逐步漂移

### 8.2 控制策略

- 继续冻结 V1 语言与字节码，不在本阶段扩语法
- 先做串口下载最小协议，再考虑 BLE/WiFi
- 持续使用同一 `main.hbc` 做双平台日志对拍
